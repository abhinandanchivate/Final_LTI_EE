📌 What: messaging vs streaming

  • Messaging – point‑to‑point or pub/sub, discrete messages (RabbitMQ).
  • Streaming – unbounded, ordered, replayable event log.

🔹 Kafka ecosystem – core definitions

Broker – storage node

Topic – event category

Partition – ordered log

Replication – leader/follower

ISR – in‑sync replicas

Producer – publishes

Consumer – reads

Offset – position in partition

ACKs – durability level

Idempotent – no duplicates

payment events: PaymentAuthorized, PaymentCaptured, RefundInitiated

❓ Why · When · Where

😵 Pain

  • Tight coupling
  • DB bottlenecks
  • Data loss & no replay
  • Mixed batch/stream

✅ Kafka solution

  • Decoupled pub/sub
  • Horizontal scale
  • Durable log + replay
  • Unified platform

📍 When to use

High‑volume event ingestion, multiple independent consumers, exactly‑once, stream processing.
Avoid: < 100 msg/sec, simple point‑to‑point.

🗺️ Where (layers)

Edge → Streaming (Kafka) → Services / Stream processors → DB / Lake

⚙️ How to implement

  1. Cluster: brokers, KRaft/ZK, topics (partitions, RF)
  2. Producer: KafkaTemplate, acks=all, idempotence
  3. Consumer: @KafkaListener, manual commit
  4. Streams (optional): KStream, state stores
  5. Exactly‑once: transactions + read_committed

🏗️ Kafka architecture deep dive

  • Broker internals – log segments, replication manager, controller, network layer.
  • Partition – unit of parallelism, each has one leader and ≥1 followers.
  • ISR – fully caught followers; only they can become leader.
  • Leader election – controlled by controller; avoids split‑brain.
  • ZooKeeper / KRaft – metadata quorum.

🎯 Desired outcomes

  • 100k events/sec, p99 latency < 100ms
  • Zero data loss (acks=all, min.insync=2)
  • Exactly‑once per payment ID
  • Strict order per paymentId (key‑based)
  • Scale without downtime

⚠️ What can go wrong

data loss acks=0, unclean leader

duplicates no idempotence + retries

out‑of‑order max.in.flight>1 without idempotence

offset issues auto‑commit before process

unavailability min.insync too high

consumer lag slow processing / hot partitions

poison pill no DLT / error handling

❌ Wrong code example (data loss)

props.put(ProducerConfig.ACKS_CONFIG, "0"); // no ack
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, false);

❌ Wrong code (duplicates)

props.put(ProducerConfig.RETRIES_CONFIG, 10);
// idempotence false -> retries may cause duplicates

🔍 Why it fails (root causes)

  • Data loss: acks=0/1 + leader crash before replication + unclean election
  • Duplicates: producer retries & broker accepted first (no idempotent)
  • Out‑of‑order: multiple in‑flight + retries ⇒ later message arrives first
  • Offset issues: misunderstanding of commit timing (poll vs process)
  • Unavailability: all replicas out of ISR, or min.insync > ISR size
  • Lag: under‑provisioned consumers, uneven key distribution
  • Poison pill: no schema validation, missing error handler

✅ Correct approach – mitigation patterns

  • Producer: acks=all, enable.idempotence=true, max.in.flight=5
  • Consumer: manual commits, store processed IDs (idempotent)
  • Dead‑letter topic for unrecoverable errors
  • min.insync.replicas=2, replication.factor=3
  • Monitor lag, partition skew, ISR
  • Use schema registry (Avro) + compatibility checks

📐 Key principles

  • CAP – Kafka is AP, but can be tuned for strong consistency within ISR
  • Idempotency – producer id + sequence number; consumer dedupe via DB
  • Ordering – per partition guaranteed; use same key for same paymentId
  • Exactly‑once – idempotent producer + transactions + read_committed
  • Log compaction – keep latest value per key (for snapshots)

💻 Correct implementation (Java / Spring)

application.yml (producer)

spring.kafka.producer:
  acks: all
  retries: 10
  enable-idempotence: true
  max-in-flight-requests-per-connection: 5

Consumer with manual commit & idempotency

@KafkaListener(topics = "payment-events")
public void consume(ConsumerRecord<String, PaymentEvent> record, Acknowledgment ack) {
    if (isProcessed(record.value().getPaymentId())) {
        ack.acknowledge(); // skip duplicate
        return;
    }
    processPayment(record.value());
    markProcessed(record.value().getPaymentId());
    ack.acknowledge(); // commit after success
}

idempotency via DB unique constraint; manual ack ensures at‑least‑once + dedupe.

🔁 Exactly‑once (transactional outbox)

@Transactional("kafkaTransactionManager")
public void processAndSend(PaymentEvent event) {
    database.updateBalance(event);   // same transaction
    kafkaTemplate.send("payment-events", event);
    // if DB fails, Kafka transaction rolls back
}

Requires transactional.id and consumer isolation.level=read_committed.

🧯 Error handler with Dead Letter Topic

@Bean
public CommonErrorHandler errorHandler(KafkaTemplate<?,?> template) {
    return new DefaultErrorHandler((record, exception) -> {
        // send to DLT after retries exhausted
        template.send("payment-events-dlt", record.value());
    }, new FixedBackOff(1000L, 3));
}

🧠 Common mistakes (senior anti‑patterns)

  • Default producer settings (acks=1, no idempotence)
  • Auto‑commit with at‑least‑once business logic
  • Ignoring rebalance listeners → duplicate on rebalance
  • No poison pill handling → consumer stuck
  • Assuming partition count is static
  • Overusing transactions (performance hit)
  • Poor key choice (hot partitions / out‑of‑order)
  • No lag monitoring
  • Misconfigured max.poll.records / interval
  • Blocking send with .get()
